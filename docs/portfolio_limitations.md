# Portfolio Limitations & Scope

> **Phase 15 deliverable.** An honest statement of what this project **is**, what it is
> **not**, and what a production build would add. Being explicit about scope is itself part of
> good engineering practice — and it keeps the portfolio credible.

This is a **design-and-implementation reference** built to demonstrate cloud data-warehouse,
dimensional-modeling, and compliance-analytics skills on Snowflake. It is intentionally
**portfolio-safe**: synthetic data, conservative compute, and no secrets.

---

## 1. Data is synthetic

- Every value — players, accounts, transactions, amounts, market/GGR figures — is
  **fabricated and illustrative**, generated by
  [`01_ingestion/05_generate_synthetic_data.sql`](../snowflake/01_ingestion/05_generate_synthetic_data.sql).
- It represents **no real individuals, customers, operators, or markets**, and is not derived
  from any company-internal or regulator data.
- The data is deliberately shaped so the AML typologies fire; **alert volumes, escalation
  rates, and GGR are demonstration artifacts**, not realistic base rates. Real transaction
  monitoring produces very different (and heavily tuned) alert distributions.

## 2. Not executed by the author against live Snowflake

- All SQL was authored and **statically reviewed**, but **not run** against a live Snowflake
  account by the author (no account was used to build it).
- Therefore **no test is claimed as "passed."** Each script ships with end-of-file validation
  queries; **their output is the source of truth** when you run the platform yourself
  (see [`deployment_guide.md`](deployment_guide.md)).
- The validation and reconciliation scripts in `07_data_quality/` are written to be run
  manually and reviewed, not as an automated CI gate.

## 3. Modeling simplifications

- **SCD Type 1** on `DIM_PLAYER` / `DIM_ACCOUNT` — dimensions hold the *current* attribute
  values. A production build would add **SCD Type 2** history (risk-rating and KYC changes
  over time); the roadmap for this is in [`data_model.md`](data_model.md).
- **Dimension attributes are synthesized deterministically** (via `HASH()`), not sourced from
  a real KYC/CRM system.
- **Informational foreign keys** — Snowflake does not enforce FK constraints; they document
  intent and support query optimization but are not integrity-enforced.
- The **grain firewall** on `FACT_MARKET_PERFORMANCE` (market data joins only `DIM_DATE`) is a
  deliberate constraint, not a limitation — but it means market and transaction facts are
  never blended below the pre-aggregated reporting layer.

## 4. AML / STR logic is rule-based and illustrative

- The **11 typologies** are transparent, threshold-based SQL rules with an explainable additive
  score. This is intentional (auditability), but it is **not** a statistical or ML detection
  model, and thresholds are illustrative rather than tuned to a real risk appetite.
- The **STR workflow** (statuses, analyst assignment, SLA days, investigation duration) is
  **simulated deterministically**, not driven by real case-management events. "Open" case
  ages are computed against a simulated as-of date.
- Nothing here constitutes regulatory advice or a compliant filing process; it models the
  **analytics layer** that would support a real AML program, not the program itself.

## 5. Platform scope

- **Single account / single region**; no multi-region, replication, or disaster-recovery
  design.
- **Batch-oriented.** The Streams & Tasks in `08_automation/` demonstrate incremental
  processing but are **optional and left suspended**; there is no true real-time/streaming
  ingestion.
- **Governance is demonstrated, not hardened.** Masking, row-access policies, RBAC, tags, and
  Time Travel are implemented to show the pattern; a production deployment would add secrets
  management, SSO/SCIM, network policies, full audit logging, and change management.
- **Snowpark** is a single illustrative feature-engineering script, not a productionized
  scoring pipeline.
- **The Power BI `.pbix` is not built** — Phase 14 delivers the connection guide, semantic
  model, DAX measures, and a page-level dashboard specification; assembling the report file is
  a documented manual step.

## 6. What a production build would add

- SCD Type 2 history and a real KYC/CRM source feed.
- Tuned, back-tested detection (including statistical/ML models) with alert-quality metrics.
- Integration with a real case-management system and regulator filing channel.
- CI/CD for the SQL (dbt or native), automated tests, and environment promotion (dev → prod).
- Full observability: query/cost monitoring, data-freshness SLAs, and alerting.
- Hardened security: secrets management, SSO/SCIM, network policies, and comprehensive audit.

---

## Summary

Read this project as a **skills demonstration and reference architecture** on synthetic data —
layered Snowflake engineering, dimensional modeling, explainable AML/STR logic, governance, data
quality, and BI enablement, delivered in validated phases. It is deliberately transparent about
the gap between a portfolio build and a production system, and that transparency is part of the
point.
